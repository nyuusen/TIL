# DynamoDB

- 分類的にはKVS
  - PK(+SK)というKeyに対して、データ本体をValueとして持つ
  - ただし、一般的なRedisなどのインメモリKVSと異なるのは、Valueの中身をDynamoDB側が構造として認識している点
- RDBスケールの限界を背景に誕生したもので、厳格なACIDではなく、BASE(結果整合性)を採用
- 三層構造
  - Table: データの集合体。プライマリキーのみ必須でスキーマレス
  - Item: 項目。RDBにおけるレコード。上限400KB
  - Attribute: 属性。RDBにおけるカラム。
- 無限にスケールする仕組み
  - パーティションと呼ばれる小さなシャードに分割して保持しているため
  - どのパーティション（物理ノード）に入るかはPKをハッシュ関数で計算した値で決定する
  - 同じPKを持つデータ内ではソートキーにより物理的にソートされた状態で保存される
  - →PK単体 or PK + SKの組み合わせが、その項目を一意に特定するユニークキーとなる
- 2つの読み込みのモード
  - 結果整合性のある読み込み
    - 書き込み直後に読み込むと古いデータが返る可能性がある
    - デフォルト設定はこれ
    - コスト（RCU）が半分で済む
  - 強力な整合性のある読み込み
    - 常に最新のデータを返す
    - コストは2倍、レイテンシが若干増える可能性も
- スループット管理
  - WCU Write Capacity Unit
    - 1KBの書き込み / 秒
  - RCU Read Capacity Unit
    - 4KBの読み込み / 秒
  - あらかじめ予約する「プロビジョニングモード」と使った分だけ払う「オンデマンドモード」が存在する
- インデックス
  - PKやSK以外の属性で検索する時、RDBMSのように自由に検索すると、全データをスキャンすることになり、パフォーマンスが悪化するので、インデックスを使用する
  - GSI Global Secondary Index
    - 全く別のPK/SKを設定し、データを別のビューとしてコピーを保持
    - ストレージ代とコストがかかる
  - LSI Local Secondary Index
    - PKは同じでSKだけを変えたい場合に使用する
    - テーブル作成時にしか作れない制約がある
