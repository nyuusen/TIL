# DynamoDB

## 基本

- 分類的にはKVS
  - PK(+SK)というKeyに対して、データ本体をValueとして持つ
  - ただし、一般的なRedisなどのインメモリKVSと異なるのは、Valueの中身をDynamoDB側が構造として認識している点
- RDBスケールの限界を背景に誕生したもので、厳格なACIDではなく、BASE(結果整合性)を採用
- 三層構造
  - Table: データの集合体。プライマリキーのみ必須でスキーマレス
  - Item: 項目。RDBにおけるレコード。上限400KB
  - Attribute: 属性。RDBにおけるカラム。
- 無限にスケールする仕組み
  - パーティションと呼ばれる小さなシャードに分割して保持しているため
  - どのパーティション（物理ノード）に入るかはPKをハッシュ関数で計算した値で決定する
  - 同じPKを持つデータ内ではソートキーにより物理的にソートされた状態で保存される
  - →PK単体 or PK + SKの組み合わせが、その項目を一意に特定するユニークキーとなる

### 2つの読み取りモード

- 結果整合性のある読み取り
  - 書き込み直後に読み込むと古いデータが返る可能性がある
  - デフォルト設定はこれ
  - コスト（RCU）が半分で済む
- 強力な整合性のある読み込み
  - 常に最新のデータを返す
  - コストは2倍、レイテンシが若干増える可能性も
- どちらの読み取りモードを使うかは、GetItemなどの読み取り用メソッド呼び出し時に指定する
  - 指定なしのときはデフォルト＝結果整合性
  - 本来リクエストを受け取るリクエストルータは、3台のノードのうち1台をランダムに選出するが、強整合性読み取りの時はLeaderに対して読み取りリクエストを送信する（ことで、強整合性での読み取りを実現している）
- GSIを指定した呼び出しでは、結果整合性のみとなる
  - 内部構造的に物理的に別の場所にデータを複製しているので、書き込みと読み取りを同期できないため
  - LSIは、ベーステーブルと同じパーティションにあるので、強整合性読み取りが可能

### スループット管理

- WCU Write Capacity Unit
  - 1KBの書き込み / 秒
- RCU Read Capacity Unit
  - 4KBの読み込み / 秒
- あらかじめ予約する「プロビジョニングモード」と使った分だけ払う「オンデマンドモード」が存在する

## キー

- PK＋SK
  - 各パーティションには、PKとSKを結合したキーを元にソートされた状態で保持される
    - SKの文字コード順

## インデックス

- PKやSK以外の属性で検索する時、RDBMSのように自由に検索すると、全データをスキャンすることになり、パフォーマンスが悪化するのでインデックスを使用する
  - 設計時は検索ユースケースを想定することが大事
- GSI Global Secondary Index
  - 全く別のPK/SKを設定し、データを別のビューとしてコピーを保持
  - ストレージ代とコストがかかる
- LSI Local Secondary Index
  - PKは同じでSKだけを変えたい場合に使用する
  - テーブル作成時にしか作れない制約がある

## QueryとScan

- PKを指定しなかった場合はScan＝全データを走査するとなってしまい、パフォーマンス面やコスト面でかなり悪い影響が出てしまう
  - 課金対象は読み取ったデータ量（スキャン量）に対してなので、コストが増える
- 対策としては、GSIを作るか、そもそも検索側（アプリ）を考え直すか..
  - つまり最初のPK設計がかなり大事であることがわかる

  - 正規化や結合の話
    - 「JOINs の必要性をなくすことが NoSQL データモデリングの中核となります。これが、Amazon.com をサポートするために DynamoDB を構築した理由であり、DynamoDB があらゆる規模で一貫したパフォーマンスを提供できる理由」
    - 「データを正規化することでディスクに保存されるデータの量は削減されますが、多くの場合、パフォーマンスに影響する最も制約の厳しいリソースは CPU 時間とネットワーク遅延です。」
      - これはJOINの処理はCPUで行われているため
    - 「DynamoDB は、JOINs を削除し(かつデータの非正規化を促進し)、データベースアーキテクチャを最適化して、項目に 1 回のリクエストでアプリケーションクエリに完全に応答します。」
  - トランザクションの話
    - 「RDBMS の速度を低下させるもう 1 つの要因は、正規化されたスキーマに書き込むためにトランザクションを使用することです。」
    - 「DynamoDB にトランザクションを実装すると、RDBMS で一般的に見られるスケーリングの問題が防止されます。DynamoDB では、トランザクションを 1 回の API 呼び出しとして発行し、その 1 回のトランザクションでアクセスできる項目の数を制限することで、これを行っています。」
