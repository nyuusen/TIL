# Go言語100Tips

## 第2章 コードとプロジェクト構成

### インターフェイス

- インターフェイスが大きくなればなるほど(メソッドを定義すればするほど)抽象化は弱くなる
  - 多用は良くない
- 使い所
  - 共通の振る舞い
    - sortパッケージの例
      - 中身のソートアルゴリズムを気にすることなく済む
  - 具体的な実装との分離
    - インターフェイスに依存するコードを書くことで実装を別のものに置き換えられる
  - 振る舞いの制限
    - あえてインターフェイスを絞る
      - 例に上げられていたのはConfigのGetのみをインターフェイスとしたものを使わせることにより、Setさせない

#### インターフェイス汚染

- よく過剰にインターフェイスが使われているのを見かけるが、あれはGoのやり方ではない
  - おそらくJavaやC#を経験したプログラマが、具象型を定義する前にインターフェイスを作成するのが自然だと思ったのでしょう
- **インターフェイス・抽象化は作成するものではなく、発見するもの**
  - 抽象化をすぐ行う必要がないのであれあば、コードの中で抽象化を行うべきではない
  - 必要だと予想される時に作成するのではなく、必要になった時に作成すべき
- インターフェイスを過剰に使うことの問題点は、コードの流れを複雑にしてしまう
  - また、インターフェイスを介してメソッドを呼び出すと、性能上オーバーヘッドが発生することもある（すごい小さいものだが）
  - インターフェイスが指す具象型を見つけるために、ハッシュテーブルのデータ構造を検索する

#### 生産者側のインターフェイス

- インターフェイスを定義する場所
  - 具体的な実装と同じ場所にすべきではなく（これもJavaやC#の慣習）、消費者側にあるべき
    - そのインターフェイスを使用したいクライアントが複数いる場合はどうするのか？気になる
      - A.下でも書いているが、そのクライアントに必要なインターフェイスのみを各々定義する（これはこれで同じようなインターフェイス定義が各所に書かれる気がするが...）
  - 抽象化は発見されるものなので、生産者側が行うことではなく、必要かどうかの判断やその必要性に最適な抽象化レベルを決定するのはクライアントである
  - Goは暗黙的にインターフェイスを満たせる言語なので、クライアント側で必要なメソッドのみを持つインターフェイスを定義(private)するべき
    - インターフェイス分離の原則（SOLID原則のI）
  - ただし明確に必要性がある場合のみは生産者側に書く
    - その時、インターフェイスをできるだけ小さくして再利用性を高め、簡単に他と組み合わせて構成可能になるように努力すべき

#### インターフェイスを返す

- 関数のシグネチャを設計する時、インターフェイスを返すか具体的な実装を返すか？
  - **インターフェイスを返すのは、Goの悪い慣習とみなされる**（まじか！）
  - これは、クライアント側にインターフェイスを定義する前提で、特定のインターフェイスを返す＝消費者側に依存する＆インターフェイスが制限される
    - こうしてしまうと、他の新たな消費者が出てきた時に、別のクライアントで定義されているインターフェイスを使うことになりかねない
- 「自分のやることは保守的に、他人から受け入れることは寛容に」をGoに当てはめると、
  - **インターフェイスの代わりに構造体を返す＆可能な限りインターフェイスを受け付ける**
    - (個人の感想)これは感じることがあって、S3 Putする関数の引数に渡すPut対象を[]byteにしている関数が実装されており、非常に不便に感じた（結局io.Readerに修正した）

#### ジェネリクスの使い所

- 型引数を与えることをインスタンス化(instantiation)と呼び、その作業はコンパイル時に行われる
  - この作業により、型安全性が言語の基本機能の一部として維持され、実行時のオーバーヘッドを避けている
  - 逆にanyのままだと、型安全性はないし、実行時にオーバーヘッドがかかるということの裏返し
- Goのmapのキー型は、等価比較==などが可能な、comparableというインターフェイスを保証する必要がある
  - これは内部実装がハッシュテーブルであり、キーのハッシュ値を計算した後、ハッシュ値比較を行うためである
- ジェネリクスを使わない方が良いケース
  - そもそもジェネリクスは必須ではないし、Goプログラマは10年以上ジェネリクスなしで開発してきた
  - 使うべきではない場合はインターフェースを使うべきではない時と類似性がある
  - まぁとりあえず必要になったら使うというスタンスでよさそう

#### 型埋め込みで起きうる問題を理解していない

- Goでは構造体フィールドが名前なしで宣言されていると埋め込み＝embededとなる
  - 埋め込まれた型のフィールドやメソッドをプロモートできる
- インターフェースも埋め込める
- 起きうる問題としては、埋め込んだ場合は外からそのメソッド等にアクセスできる
  - アクセスさせたくないなら、小文字はじまりで名前付してプライベートフィールドにするべき
- 結論として、型埋め込みが必要になることは稀であり、主には振る舞いのプロモートするため＝利便性のために使われる

#### 関数オプションパターンを使わない

- Config構造体を渡す
  - フィールドのゼロ値が意図した値かを判断するのにポインタをセットする必要がある
  - デフォルトを使いたい場合は空の構造体を渡す必要があり、見栄えが悪い
- ビルダーパターン→Goにはあまりフィットしない
  - GoF

    ```
    NewServerBuilder().
    Addr(":8080").
    Timeout(10 * time.Second).
    MaxConn(100).
    Build()
    ```

  - Goは戻り値でエラーを返すので、最後のBuild()でまとめて返すか、途中でpanicを起こすしかない
- 関数オプションパターン
  - aws-sdkとかでよく見るやつ
  - 慣習的にWithで始まる
  - 可変個の引数オプションをもらうことで、オプションが増えても呼び出し元もプロバイダ側のNew関数を変更する必要がなくなる
  - Goではよく使われる
- ユーティリティパッケージ
  - utilsやsharedというパッケージ名は何の見識も与えない無意味な名前
  - セットデータ構造の例だとstringsetのようなパッケージ名にしてNew()を定義したりする
  - そこから生やすソート関数などは、専用の型を作りそのレシーバ関数を定義することで、stringsetパッケージへの参照はNewの一度だけになる
- オーバーフロー
  - コンパイル時に判断できるものはエラーとなるが、実行時のオーバーフローはエラーも出力されないしパニックにもならない
- スライスの長さと容量を理解していない
  - int、長さ3、容量6のスライスを作ると先頭から3要素がゼロ値である0、後半3要素は割り当てられているがまだ使われていないという状態
    - これに対して、s[4]のようにアクセスするとパニックが発生する
    - 要素を追加する場合はappendを使用する
  - 容量がいっぱいになると、内部的に容量を2倍にした新たな配列に要素をコピーする
    - 古い配列はヒープに割り当てられててもう使われないならGCによってクリーンされる
  - スライスの長さはスライスで利用できる要素数で、容量は基底配列の要素数
- 非効率なスライスの初期化
  - ある配列から異なる配列を生成する関数を例に、長さ指定なし、長さ指定、容量指定のベンチマークを比較すると、長さ指定(要素はインデックスアクセスで設定)が最も早い
  - 容量指定はappendの若干のオーバーヘッドがかかる
  - ただし、インデックスアクセスは可読性の観点で、あえて容量指定を選択することもある
- nilスライスと空スライスを混乱する
  - 割り当ての違い
  - 特に理由もない割り当てを防ぐ意味でnilスライスを返すべき
- スライスとメモリリーク
  - スライスから再スライスを行っても(巨大なスライスの先頭数要素のみをスライス化して取得する)基底配列は残るのでメモリに残り続ける、結果的にメモリリークさ！？
  - GCはあくまで到達可能性を見るので、基底配列全体が使用中とみなされ、全体がメモリに残る
  - 代替としてはappendする、copyする
  - **経験則として、大きなスライスや配列をスライス化すると、メモリを大量に消費する可能性があることは覚えておく**
- マップの概念
  - キーバリューの順不同のコレクション
