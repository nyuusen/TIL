[bohnen/bbf-newsql: 翔泳社「わかばちゃんとつくって、壊して、直して学ぶ NewSQL入門」サポートページ](https://github.com/bohnen/bbf-newsql)

## 4章 NewSQLの処理の流れを見てみよう

- TiDB: コンピュート層
  - クライアントとの接続確立
  - SQLの文法やスキーマ存在チェック
  - SQLをASTに変換し、実行計画を作成する
- TiKV: ストレージ層
  - ストレージを管理し、問い合わせのAPIを提供
- PD: コントロールプレーン層（制御層）
  - 各サーバーの配置を管理し、クラスター全体に関係するメタデータや現在の状態、負荷状況を収集・管理する
    - 問い合わせに行くならあのTiKVノードがおすすめやでを言ってくれる
- この分離しているアーキテクチャ自体は最近のDBに共通するもので、それぞれ独立してスケールアウトできたり、複数の性能が異なるストレージを混在させたりできる

### 参照クエリ

- クライアントからクエリ送信
  - mysqlプロトコルを利用
- TiDBサーバー
  - 構文チェックなどからAST変換
  - ASTに対して処理が効率的に実行されるように最適化
  - 最終的にTiKVで実行可能な一連の処理の流れが作成される（実行計画と呼ばれ、EXPLAINなどで確認できる）
- TiKVからデータ取得
  - TiDBサーバーからTiKVサーバーのデータ取得APIを呼び出し
    - WHERE句の内容によっては複数回TiKVへの問い合わせが発生する（
      - これが少なくなるようにTiKVサーバー単位でまとめて呼び出せるように最適化が行われる
    - 複数台への問い合わせは並列で行われる
  - TiDBとTiKVの間の通信は、gRPC上で動く
- PD
  - TiKVサーバーはデータの保存の単位としてリージョンと呼ばれるデータの塊
    - 具体的には複数のレコードをPKの一定範囲で区切る
    - 1つのリージョンはデフォルトで96MBgh repo clone nyuusen/bbf-newsql
    - 1つのリージョンは3つのレプリカ＝コピーを持っており、Raftという合意形成アルゴリズムを使って整合性を担保している
      - この3つのうちの1つがLeaderとなり、2つはFollowerとなる
        - Leaderが窓口で、FollowerはLeaderから送られてくるログを保存する
        - なぜ3つのコピーがあるのに、わざわざPDに教えてもらって、Leaderにアクセスをする必要があるのか？
          - 高速性と一貫性: コピーされてノードから取り出そうとするとコピーの時間だけラグが発生するので、書き込まれたTiKVノードから取り出すのが最も速く返せる
          - なので、TiKVではリージョンごとにLeaderを決めて、読み書き両方を担当させることで、一貫性のあるデータを返せるようにしている
          - ただ読み書き両方の処理を受け付ける分負荷は高まりレスポンスは劣る
  - TiDBは、このリージョン単位でTiKVにアクセスしており「どのTiKVにアクセスすれば良いか」の情報を与えているのがPDの仕事

### 更新クエリ

- 基本は参照クエリと同じ
- TiDBサーバでのクエリ解析とトランザクション管理
  - トランザクション管理を行う
    - 複数のTiKVサーバーへの書き込みの際に全てが確実に書き込まれることを保証する役割を担う
      - いずれかの処理が失敗したらロールバックを行う
    - トランザクション管理には2フェーズコミットという手法をとっている
      - 最初のフェーズでコミットの準備（TiDBの場合は更新対象のロックを取得）をし、次のフェーズで確定として実際の書き込みを行う
- TiKVへのデータ書き込み
  - PDサーバーからLeaderを取得し、Leaderに対して書き込みを指示する
    - Leaderは書き込み操作そのものを複数のTiKVサーバーにコピーする（デフォルトでは計3サーバーに同じデータがコピーされた状態になる）
      - このコピーにRaftというアルゴリズムを使用している
- PDサーバーの役割
  - 参照クエリと同様にLeaderを教えてくれる
  - 他にトランザクションコミット時のタイムスタンプを記録する

### タイムスタンプの理解

- タイムスタンプは実際に時刻そのものではなくシーケンス値を使用する
- 銀行を例にすると、
  - 振り込みが先か、引き落としが先かで随分と変わる（引き落としが先になると残高エラーの可能性とか）
    - この問題に対し、リアルでは受付番号を発行することで、同時に入ってきた客にも順番をつけることができる
- まさにこの発券機の役割をしているのがPDサーバが行なっているタイムスタンプ生成
  - TiDBではSELECTやINSERTなど処理の初めに必ずこのタイムスタンプを取得している
- タイムスタンプは、自国をベースに一意のシリアル値を組み合わせたもので、TSO(TimeStampOracle)と呼ばれる
  - TSOは、分散環境においては時刻の代わりを果たすもので、論理時間と呼ばれる
- トランザクション分離レベル
  - NewSQLの多くはSnapshot Isolationというもので、これはトランザクションの間は常に同じデータを読み込むことができる
    - 処理を開始した時点のスナップショットを見るような感じでデータを読み取る
    - DBはタイムスタンプより後に確定した変更は全て無視し、Undoログから復元して見せてくれる

## 5章 1つずつ壊してみよう

- ロードバランサー
  - 製品固有のロードバランサーとしてTiProxy
    - PDが収集しているクラスターの最新情報を取得し、タイムリーなリクエストのルーティングを可能にする
    - どのTiDBサーバーへの接続情報はPDからもらう
  - 自前でクラウドのロードバランサを立てる場合、pingやヘルスチェックエンドポイントで3回連続失敗で切り離すみたいなことをするので時間を要する
  - 以下の記事を見る限りだと、TiDB直接接続だとスケールインやメンテナンス時に接続断が発生するよう
    - [レバテックのTiDBにTiProxyを導入するまでにやったこと](https://zenn.dev/levtech/articles/enabling-tiproxy)
- TiDBサーバーを全て停止するとMySQLクライアントから接続できなくなる
  - 全て停止しなくても接続していたサーバーがスケールインされると接続が切れるので、アプリケーション側でも考慮が必要な点となる
- TiKVサーバー起動中の3台のうち1台(Leader)落としてみる
  - 3台のうち1台(Leader)を落としてもクエリ実行は成功する
  - TiKVサーバーは相互にハートビートを送っていて、リーダーの停止が確認できると、フォロワーノードのいずれかがリーダーに昇格して処理を続行する
    - 手動のクエリ実行だと↑を表現するのは中々難しいが、今回はTiDBからTiKVへのリクエストでリトライが行われたので、TiKVリーダーノードの停止は影響を受けない（リトライの分だけレイテンシは若干増加するが）
- さらにTiKVサーバーを残2台のうち1台を落としてみる
  - クエリは失敗するようになる
    - これはネットワーク分断障害と区別がつかないためこのような挙動となる
      - 自分が最新なのか、はたまたネットワーク分断で自分だけが切り離された古い情報なのかの判断がつかない
      - Raftアルゴリズムによる多数決の原理が関係している
- PDを停止してみる
  - PDもTiKVと同様に2台だと処理継続できるが、1台では不可
  - PDでもRaftアルゴリズムが使用されている

## 6章 NewSQLの仕組み・アーキテクチャを理解しよう

- コンピューティング層とストレージ層の分離

### コンピューティング層(TiDBノード)

- SQLパースと解析
  - クライアントから受け取ったSQL文を、字句解析(Lexer)と構文解析(Parser)を経て、AST(Abstract Syntax Tree)へと変換される
    - 字句解析(Lexer)
      - SELECTなどの最小単位のトークンに分割
    - 構文解析(Parser)
      - 切り分けたトークンをSQLの文法規約に照らし合わせ、親子関係を持つツリー構造(AST)に組み立てる
      - ここでSELECTの後にFROMがないみたいな構文エラーが発生する
- 最適化と実行計画の生成
  - ASTに対して、論理最適化・物理最適化が適用され、最適な実行計画を生成する
  - 論理最適化
    ```
    ここからがTiDBの腕の見せ所です。SQLの中身を「結果を変えずに、より効率的な手順」に書き換えます。

    - 述語プッシュダウン (Predicate Pushdown):
    JOIN した後に WHERE で絞り込むのではなく、JOIN する前に各テーブルで絞り込みを終わらせるよう順序を入れ替えます。転送データ量を最小化する、分散DBで最も重要な最適化です。

    - カラムプッシュダウン (Column Pruning):
    SELECT * と書いてあっても、実際に後の工程（JOINやORDER BY）で使わないカラムは、ストレージ（TiKV）から読み出さないように削ります。

    - サブクエリの相関解消 (Max/Min Eliminationなど):
    サブクエリを可能な限り JOIN に書き換えます。ループ（依存関係）をなくし、セットベースの処理に変えることで並列実行しやすくします。
    ```
  - 物理最適化
    ```
    論理的な手順が決まったら、次は**「どのインデックスを使い、どのアルゴリズムで結合するか」**を具体的に決めます。こちらは「コストベース最適化（CBO: Cost-Based Optimization）」と呼ばれます。

    TiDBは以下の要素を「コスト」として数値化し、合計が最小になるルートを探します。

    コストの構成要素:
    - CPU: データの比較やハッシュ計算の負荷。
    - Memory: ソートやハッシュテーブル作成に使うメモリ量。
    - Network: TiKVからTiDBへデータを送る通信負荷。
    - Disk I/O: ディスクから読み出すブロック数。

    具体的な選択肢の例：
    - Index Selection: WHERE a = 1 AND b = 10 のとき、インデックスAを使うか、Bを使うか、あるいはフルスキャン（TableScan）の方が速いかを、統計情報（データの分布）から予測される「読み取り行数」を元に判断します。
    - Join Algorithm Selection:
    - Hash Join: 片方のテーブルがメモリに収まるサイズなら、ハッシュテーブルを作って一気に結合します（大容量向け）。
    - Index LookUp Join: 外側のテーブルが小さく、内側のテーブルにインデックスがある場合に、1行ずつインデックスを引いて結合します（小〜中容量向け）。
    ```
  - この作業により、実際のデータアクセスや結合処理が効率化され、クエリの実行時間を短縮できる
- ステートレス
  - SQL変換のみでデータを保持したりしない、意図的にステートレスな設計となっている
  - なのでクライアントからの大量接続に対して容易にスケールアウトが可能

### ストレージ層(TiKVノード)

- データの永続化
  - ストレージエンジンはRocksDBを利用
  - Write-Ahead Log(WAL)やMemTable,SSTファイルといった仕組みでデータの永続性を保証する
    - RocksDBは、LSM-Tree
    - WAL: ディスク上のログファイルにこれから書くデータを追記(AppendOnly)
      - MemTableが揮発したとしてもリカバリ可能
    - MemTable: WALに書くと同時に、メモリ上のデータ構造であるMemTableにデータを展開する
      - SkipListという構造で、データが常にソートされた状態
    - SST(Sorted String Table)ファイル: MemTableがいっぱいになると一気にディスクに書き出す
      - 増えてくると裏側でコンパクション処理が走り、古いデータや削除フラグが立ったデータを掃除していく
    - →これらLSM-Tree（Log-Structured Merge-Tree）というアーキテクチャである
  - 分散データ管理
    - Raftアルゴリズムで一貫性や冗長性を担保
  - 分散トランザクションとMVCC
    - 複数ノードにまたがるトランザクション処理を実現する
    - 同時実行性が高まり、システム全体のパフォーマンスが向上する

### インターフェイスとデータ分散の仕組み

- 物理的なデータは自動的に複数のノードに分散されるシャーディングという仕組みにより、データの負荷分散とスケーラビリティを担保する

### 冗長性と可用性の担保

- TiKVではリージョンというデータ単位で、複数のPeer(対)を構成し、各ノードに配置される
  - 1台のノードが障害を起こしても、システム全体として可用性が担保される

### NewSQLのメリット

- スケールアウトのしやすさ
  - 負荷に応じて各層を柔軟に増減できる
  - 例えばクライアント接続が増大したらTiDBノード、データ量が増えたらTiKVノードを追加するのような
- 書き込み性能
  - ストレージ層がKVSなので書き込み処理が効率化
  - 分散トランザクションと冗長化されたデータ配置により、大量のデータが同時に書き込まれる状況でも高いスループットを実現
- 障害時の自動復旧動作
  - 一部のノードが停止してもサービス継続できる
- 透過的な分散処理
  - 分散処理、シャーディング、トランザクション管理などの内部的な複雑さは、内部で自動的に処理されるので、ユーザーは従来のRDBと同じ感覚でデータ操作が可能

### TiDBノード

- クライアント接続の処理
  - プロトコルレイヤーにて、MySQLプロトコルで接続してきたユーザー認証・権限を確認
- SQL解析とコンパイル
  - SQL文を解析しASTを生成
  - 続いてCompiler処理
    - バリデーション: SQL文の論理的な整合性を検証（テーブル名やカラム名）
    - クエリの最適化:
      - 論理最適化
        - 意味を変えずに効率的な形へと書き換える
          - 不要カラムを削除、条件式を移動（WHEREを早い段階で適用してデータ量減らす）、サブクエリを単純なJOINに変換など
      - 物理最適化
        - 論理最適化で作られたものを基に、どのようなアルゴリズムで実行するかを決定する
        - データの分布や統計情報を考慮しながら、結合アルゴリズムやデータアクセス方法（テーブルスキャン/Indexスキャン）など
    - 実行計画の生成
      - 上記の最適化が完了すると、実際にどのようにSQLクエリを実行するかを具体的な手順を定めた実行計画を生成する
        - どのテーブルからどのような方法でデータを取り出すか結合するか
- リレーショナルデータからKVへの変換
  - Table IDとRow Key(各行のPK or 内部的な行ID)をキーとする
  - カラムの集合をバリューとする
- TiKVにおけるリージョンとデータ配置
  - 1TiKVノードないにはストレージ領域を特定のサイズで論理的に分割しており、その領域をリージョンという
    - 1つのリージョンはデフォルトで256MB
    - 1リージョンへの書き込みが256MBに達した際、別リージョンへ分割される
- SQLが実行されるまでの道のり
  - Executorの実行
    - クエリの内容に応じてシンプルなKey-Value APIと複雑な処理を行うDistSQL APIを使い分ける
  - トランザクションの処理
    - Executorにより実行計画の処理が始まると、トランザクションモジュールへの処理が移る
    - トランザクションを一意に特定するTSOをPDから取得
    - 2フェーズコミットやロックの管理を含んでいる
  - DistSQLによるクエリの実行の効率化 / KVモジュール
    - DistSQLは、TiKVに対してクエリを並列実行できるように分散させたり、一部の計算をTiDBではなく、TiKV側に処理タスクを委任したりする
  - TiKVクライアントからTiKVへの読み書きリクエストの送信
    - 実際にはこのクライアントがTiKVへの読み書きを送信する役割を担っている

### TiKVノード

- 永続的なデータとRocksDB
  - RocksDBコンポーネントをストレージエンジンとして利用し、分散KVSを構成している
    - このRocksDBがWAL,MemTable,SS Tableの実装（LSM-Treeを具現化したもの）
    - KV書き込み時の挙動
      - WALと呼ばれるログを記録するファイルに挿入される
      - MemTableに書き込みが行われる
      - 最終的にデータをディスクに追記型で格納する
    - 削除や更新も、新しいデータとして追記して書き込む
    - 追記型は、対象となるデータの場所を探す必要がないので、特に書き込み時のパフォーマンスが優れている
    - なぜメモリ上のMemTableとWALの2箇所にデータを保存するのか？
      - メモリ喪失に対して耐久性を保持するため
      - 新しい書き込みは常にMemTableにデータを挿入し、読み取りの操作はMemTableを検索してからディスク上のSSTを検索する
      - MemTableがいっぱいになると、バックグラウンドでメモリ上のImmutable MemTableに書き込まれ、その後SSTにフラッシュされる動きとなる
      - ディスクに書き込まれたら、memTableと関連するWALファイルを削除する
- LSMツリーの階層構造
  - LSMツリーにおけるディスク上のデータの考え方としては、格納されているデータを複数のレイヤー（レベル）に整理し保存している
    - 最新のデータはレベル0(L0)に格納される
    - L0には複数の小さなSSTがあり、新しい書き込みがある度にファイルが増える
    - L0のデータ量が閾値を超えるとCompactionが走り、下位のレイヤーにマージされる
    - レイヤーが下がるほど、データは統合・圧縮され、大きなSSTに整理される
  - 階層化されていることにより、
    - 全てのファイルをスキャンしなくて済む
      - でも更新・削除も追記なら結局スキャンが必要なのでは？
        - 最新1件さえ参照できれば、古い行は見なくて済むので、スキャン必要なし
    - 全てのデータを一度に並び替えるとI/Oが終わるので、局所的なコンパクションができるのはCPU負荷的に優しい
- 書き込み操作の流れ
  - WALへの追記
  - MemTableへの書き込み（ソート済み）
  - MemTableをフラッシュしSSTファイルへの変換
- 更新・削除操作の流れ
  - MemTableに更新・削除情報を記録
    - 削除なら削除フラグを記録するのみ
  - 読み取り時のフィルタリング
    - RocksDBではデータを読み取る際にまずはMemTable内の削除フラグを確認する
  - コンパクションによる最終的な削除
    - SSTにフラッシュされた後も一時的に保持されるが、その後のコンパクションの時に不要なデータがディスクから完全に削除される（ストレージ使用量を最適化）
- 読み取り操作
  - BlockCacheチェック
    - BlockCacheは頻繁にアクセスされるデータブロック(SSTのデータブロックを直接メモリに格納する)をメモリ上にキャッシュする仕組み
    - LRU(Least Recently Used)アルゴリズムで、最近使用されていないデータを破棄してメモリ最適化を行う
  - MemTableを検索
  - SSTを逆順に検索
    - 逆順＝新しい順
    - ブルームフィルタのおかげでそこにデータがあるかも/確実にないを高速に検索できる
  - ブルームフィルタ
- カラムファミリー
  - データを論理的に分割・管理する仕組み
    - 例えばすぐ消えるLockデータとそうではないUserデータとで、LSM-Treeを分けるというもの
  - データ特性に応じて、コンパクション頻度やメモリ量などを調整できる
  - 実際にTiKVでは以下の3種類のデータを分離している
    - Lock CF: トランザクションに必要なロックデータ
    - Write CF: 実際のデータの更新履歴とMVCCに必要なメタデータ
    - Default CF: ユーザーが書き込んだ実際のデータを格納するカラムファミリー

### 分散トランザクション

- 2フェーズコミット
  - PDが発行するタイムスタンプがトランザクションに割り当てられる
    - このタイムスタンプはトランザクション開始時刻とコミット時刻に使われる
- 実際のトランザクションの動き（デフォルトの悲観的モード）
  - 1.トランザクション開始
    - PDからTx開始時刻を取得
    - 悲観的Tx: DML実行時に更新対象のキーへのロックを事前に取得し、競合があれば待機orタイムアウト/エラーを返す（v3.0以降こちらがデフォルト）
    - 楽観的Tx: ロックは取得せずコミット時に競合検知を行い、競合発生時はロールバックされる（アプリ側で再試行が必要）
  - 2.データの書き込み（Prewrite）フェーズ
    - TiDBノードはレコードのキーとstart_tsを元にキーバリュー形式のデータを生成
    - その後、LockCFにロック情報を格納することで、他のトランザクションが同じキーを変更できないようにする
    - そして、DefaultCFに更新するデータを保存
  - 3.コミットフェーズ
    - PDからcommit_tsを取得
    - WriteCFにstart_tsとcommit_tsを組み合わせたデータを保存し、コミットを確定
    - その後、LockCFに削除フラグを立てる
- 分散トランザクションで信頼性を確保する仕組みとして、
  - 2PCによる原子性の確保
    - Prewriteフェーズが全て成功しないとCommitフェーズに進まない
  - プライマリロックを利用したトランザクション管理
    - プライマリロック：TiKV LeaderノードのLockCFに格納
    - クラッシュリカバリの際もプライマリロックをチェックすればロールバック処理が可能
  - MVCCによる並行処理の最適化
    - DefaultCFにデータのバージョンを管理することで、ロックが解除された後でも過去のデータに安全に参照可能

### MVCC

- 通常のロックベースのトランザクションでは更新中の読み取りをブロックしてしまい、システムの応答性が低下する
- この問題を解決するのがMVCCで、データ更新時に古いデータを削除せず、新しいバージョンのデータを追加することで、読み取りをブロックせずに古いデータを参照できる
- 基本原理
  - 各データ項目に対して複数のバージョンを保持する
  - 各トランザクションには一意のタイムスタンプが割り当てられ、読み取り操作は対応するタイムスタンプに基づいて適切なバージョンを取得する
  - これにより、読み取りと書き込み操作が互いにブロックすることなく、同時に実行可能となる
- MVCCの実装
  - TiKVはKVSなので同じキーでPutすると上書きされてしまうので、キーにコミットタイムスタンプを付与して別々のキーとして扱う
  - TiKVノードはRocksDBを使用し、WriteCFには各キーの更新履歴とコミット情報、DefaultCFには実際のデータ、LockCFにはトランザクションのロック情報を保持している
- トランザクション分離レベル
  - TiDBはデフォルトでREPEATABLE READとなっており、明示的にBEGINステートメントを実行して複数のステートメントを1つのトランザクションで実行する場合、トランザクション開始時点(start_ts)のスナップショットを使用するので同一トランザクション内では一貫した結果が得られる
- レプリカによって可用性が実現される仕組み
  - 分散システムの根本的課題はノード障害やネットワーク遅延が発生する可能性がある中で、どのようにノード間で一貫した状態を維持できるか
- ログレプリケーションの動き
  - 操作がログエントリーとして記録され、そのログがクラスター内のすべてのノードに正確に複製されるということが要
  - TiKVではデータのキーの連続性に基づいてリージョンに分割し、リージョンごとに独自のRaftグループを運用している
    - RaftグループではそれぞれのリージョンをPeerと呼ぶ
  - ログエントリーにはリージョンID+インデックスのような形で時系列順に積まれていき、操作履歴が一意に管理されている
- Raftにおけるリーダー選出とタイムアウト時の動き
  - フォロワーからリーダーから定期的に送られるハートビート（生存確認メッセージ）を受信する
  - これが一定時間受信できなくなると、各ノードは自律的な判断を迫られ、選挙タイムアウトという仕組みでクラスター内で新たなリーダーを決定するための選挙プロセスを開始する
- TiKVにおけるログレプリケーションとコミットまでの流れ
  - Proposeフェーズ
    - TiDBノードよりデータの更新や挿入の操作を、対象リージョンのリーダーに送信
    - 操作内容はログエントリーに変換される
    - ログエントリは「4_1(4はリージョンID), log { Put key=1, name='tanaka' }」のような形式
  - Appendフェーズ
    - リーダーが自らのローカルストレージのRaft logにAppendする
    - この時点ではリーダー自身が確実に保持しただけでクラスター全体で操作は確定していない
    - 後のレプリケーションと整合性チェックに利用する
  - Replicateフェーズ
    - リーダーからネットワークを介して各フォロワーに対し同一のログエントリーを送信
    - フォロワーは受け取ったエントリーを自らのストレージに保存し、リーダーへ確認応答(ACK)を返す
    - 過半数のフォロワーからACKが返ってくると、そのエントリーが安全に複製されたと判断される
  - Commitフェーズ
    - リーダーが過半数以上のACKを受領するとCommitとして確定される
  - Applyフェーズ
    - リーダーがコミット済みエントリーの内容を実際のデータストア（RocksDBのKV層）に反映
- この一連の流れにより、一度コミットされたエントリが必ず全クラスターに反映されるという強い整合性を保証する

### 分散合意アルゴリズム:Raftの仕組み

- 分散システムにおける、複数ノードが1つの値に合意する方法は長年研究されてきた結果、PaxosとRaftというアルゴリズムがある
- Raftは、Paxosの複雑な理論構造を解消するために設計されたもので、特徴は状態の更新順序やログの整合性をシンプルに管理するという点
- TiKVではキーの連続性に基づいた「リージョン」にデータを分割し、リージョンごとに独立したRaftグループとして管理する「マルチRaft」方式が採用されている
- TiKVへのデータ書き込みプロセスとRaftの挙動
  - TiDBノードがPDに書き込み対象のキーのリージョン（TiKVノード）を教えてもらう
  - 該当リージョンのリーダーに対して書き込み操作を要求
  - TiKVノードにはRocksDBインスタンスを2つ「Raft Engine(Raft Logを永続化するための専用ストレージ)」と「RocksDB KV(実際のKV)」がある
    - 内部のRaftstore poolを介して、RaftLogと実際のKVに反映する作業を並行して行う
  - このうち、まずは受信した書き込み操作をシリアル化し、RaftLogにAppend(永続化)する
    - 同時にフォロワーにレプリケートする
    - 過半数のフォロワーノードが受信できたらその操作をコミットとみなす
  - コミット済みのRaftLogエントリーを書いせし、実際のデータとしてRocksDB KVに反映させる

### TiKVにおける計算処理Coprocessorの仕組み

- TiDBノードは、必要なデータをTiKVから取得し、TiDB自身でそのデータに対する計算や集約処理を行っている
- ただこのアーキテクチャでは「TiKV側は全データをネットワーク越しにに返すので、通信のオーバーヘッドがかかる」「TiDBは大量のデータを自ら計算するためCPUリソースを多く消費する」という課題がある
- この課題を解消する仕組みとして、TiKVでも計算処理を行うコンポーネントが「Coprocessor」
- 実際の挙動としてはTiDBが実行計画に基づき、各TiKVノードにCoprocessorリクエストを送信し、TiKVノードで指定された操作（テーブルスキャンやインデックススキャンなど）を実行する

### PDノード

- クラスタの頭脳であり、クラスタ全体のメタデータ管理やリソース調整、グローバルなIDやタイムスタンプの生成を担当する
- TiKVの各ノード内のデータ配置状況やデータ分布を監視し、TiDBが正しいTiKVにクエリを送信できるようにルーティング情報を担当する
  - もし複数のTiKVノードに跨るようなデータを取得する際は、クエリの範囲をTiKVごとのタスクに分割して並列実行する
- PDは通常3ノード以上で構成され、内部に分散KVS(etcd)を組み込んでいる
  - このetcd上に、クラスターのトポロジー情報やリージョン情報、スケジューリングポリシーなどを保存し、Raftプロトコルにより自動復旧と高いデータ一貫性を保証している
- ルーティング
  - TiDBがクエリを受け付けると、内部のTiKVクライアントがまずPDにどのリージョンに属しているかを取得する
- TSO
  - グローバルタイムスタンプ
  - int64型の整数で、物理時間(Unix時間ms単位)と論理時間(1ms内に複数のTSOを生成するためのカウンタ)で構成
  - PDクラスターもRaftで運用されており、リーダーがダウンしても新たなリーダーが選出される
    - etcdを共有で管理しているので、TSOの情報（以前のリーダーが割り当てたTSO）を永続的に保存している
- スケジューリングプロセス
  - 各TiKVノードから定期的にハードビートメッセージを受信し、情報を収集する
    - 各ノードの容量やトラフィック、リージョン範囲やレプリカ分布など
  - 上記の情報とあらかじめ設定されたポリシーに基づき、スケジューリングプランを生成し、実行する
- ラベル付け
  - TiKVノードがどのラックやどのDCに置かれているかという物理的な情報をPDに教える
  - それを元にPDが最適な場所に分散するようスケジューリングを行う
- TiDBノードが止まったら？自動再接続の仕組み
  - 複数台のTiDBノードの構成の場合は、1台が落ちてもコネクションが一度切断されるだけで、別のTiDBノードに接続が切り替わる動きをする
  - TiDBノード自体はステートレスで、セッション情報やデータそのものは残らない
  - TiProxyや多くのMySQLクライアントライブラリが備える自動の再接続機能によって、一時的にエラーが起きても再接続がシームレスに行われる
  - ただし、トランザクション途中でTiDBノードが停止されるとロールバックされるので、SQL発行のリトライは必要になる
- TiKVノードが止まったら？リーダーがダウンしても動き続ける仕組み
  - 3台構成の場合、2台が生きていれば新たなリーダーを選出可能であり、クラスター全体の可用性が保たれる
  - TiDBノード側では、リーダーから応答が返ってこない場合、自動でリトライ処理を行ってくれる
  - ただし、3台中2台が落ちてしまうと、リーダーを決めるための多数決が成立しないので、手動でPDやTiKVを再起動するなどの手順を踏む必要がある
- PDが止まったら？TSOの重要性
  - これも3台中2台が停止すると、PD server timeoutが返り、TSO取得ができなくなる

## 7章 クラスター運用時に考えること

- レスポンスタイムと耐障害性のトレードオフ
  - 多くのNewSQLでは、このトレードオフをユーザーが選択できるようになっている
  - TiKVのリージョンの配置はPDが管理しているが、リージョンのコピーは必ず別のAZに配置されるように指示することは可能（反対にレスポンスタイムを優先して同一地域に配置するようにすることも可能）
  - TiDB Cloudには「性能を重視してユーザーごとに専用のVPC,サーバを利用するDedicated」と「コスパ重視のサーバは全ユーザーで共有するマルチテナントモデルのServerless」の2つが存在する
  - TiDB Cloud Dedicated
    - 必ずAZを跨るような構成でサーバーを配置するので高い耐障害性を確保
  - TiDB Cloud Serverless
    - 同じAZに配置（マルチAZオプションはある）
- コネクションの切断
  - バージョンアップはマネージドサービス側でクラスタ内でローリングアップデートされる（多くのNewSQLがそう）
  - Chapter5の実験で、PDとTiKVの停止はTiDB側でリトライされるので、ユーザーセッションに影響はないことは確認済み
  - だがTiDBが再起動すると、そのTiDBに接続していたセッションが切断される
    - この影響を軽減するために、TiDB以外のコンポーネントからローリングアップデートが開始されるが、最後にTiDBを更新する際にコネクション切断が発生する
  - TiDBサーバー自身はGracefulShutdown（終了前に待機時間がありセッションの終了まで待つ）
