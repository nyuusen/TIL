[bohnen/bbf-newsql: 翔泳社「わかばちゃんとつくって、壊して、直して学ぶ NewSQL入門」サポートページ](https://github.com/bohnen/bbf-newsql)

## 4章 NewSQLの処理の流れを見てみよう

- TiDB: コンピュート層
  - クライアントとの接続確立
  - SQLの文法やスキーマ存在チェック
  - SQLをASTに変換し、実行計画を作成する
- TiKV: ストレージ層
  - ストレージを管理し、問い合わせのAPIを提供
- PD: コントロールプレーン層（制御層）
  - 各サーバーの配置を管理し、クラスター全体に関係するメタデータや現在の状態、負荷状況を収集・管理する
    - 問い合わせに行くならあのTiKVノードがおすすめやでを言ってくれる
- この分離しているアーキテクチャ自体は最近のDBに共通するもので、それぞれ独立してスケールアウトできたり、複数の性能が異なるストレージを混在させたりできる

### 参照クエリ

- クライアントからクエリ送信
  - mysqlプロトコルを利用
- TiDBサーバー
  - 構文チェックなどからAST変換
  - ASTに対して処理が効率的に実行されるように最適化
  - 最終的にTiKVで実行可能な一連の処理の流れが作成される（実行計画と呼ばれ、EXPLAINなどで確認できる）
- TiKVからデータ取得
  - TiDBサーバーからTiKVサーバーのデータ取得APIを呼び出し
    - WHERE句の内容によっては複数回TiKVへの問い合わせが発生する（
      - これが少なくなるようにTiKVサーバー単位でまとめて呼び出せるように最適化が行われる
    - 複数台への問い合わせは並列で行われる
  - TiDBとTiKVの間の通信は、gRPC上で動く
- PD
  - TiKVサーバーはデータの保存の単位としてリージョンと呼ばれるデータの塊
    - 具体的には複数のレコードをPKの一定範囲で区切る
    - 1つのリージョンはデフォルトで96MBgh repo clone nyuusen/bbf-newsql
    - 1つのリージョンは3つのレプリカ＝コピーを持っており、Raftという合意形成アルゴリズムを使って整合性を担保している
      - この3つのうちの1つがLeaderとなり、2つはFollowerとなる
        - Leaderが窓口で、FollowerはLeaderから送られてくるログを保存する
        - なぜ3つのコピーがあるのに、わざわざPDに教えてもらって、Leaderにアクセスをする必要があるのか？
          - 高速性と一貫性: コピーされてノードから取り出そうとするとコピーの時間だけラグが発生するので、書き込まれたTiKVノードから取り出すのが最も速く返せる
          - なので、TiKVではリージョンごとにLeaderを決めて、読み書き両方を担当させることで、一貫性のあるデータを返せるようにしている
          - ただ読み書き両方の処理を受け付ける分負荷は高まりレスポンスは劣る
  - TiDBは、このリージョン単位でTiKVにアクセスしており「どのTiKVにアクセスすれば良いか」の情報を与えているのがPDの仕事

### 更新クエリ

- 基本は参照クエリと同じ
- TiDBサーバでのクエリ解析とトランザクション管理
  - トランザクション管理を行う
    - 複数のTiKVサーバーへの書き込みの際に全てが確実に書き込まれることを保証する役割を担う
      - いずれかの処理が失敗したらロールバックを行う
    - トランザクション管理には2フェーズコミットという手法をとっている
      - 最初のフェーズでコミットの準備（TiDBの場合は更新対象のロックを取得）をし、次のフェーズで確定として実際の書き込みを行う
- TiKVへのデータ書き込み
  - PDサーバーからLeaderを取得し、Leaderに対して書き込みを指示する
    - Leaderは書き込み操作そのものを複数のTiKVサーバーにコピーする（デフォルトでは計3サーバーに同じデータがコピーされた状態になる）
      - このコピーにRaftというアルゴリズムを使用している
- PDサーバーの役割
  - 参照クエリと同様にLeaderを教えてくれる
  - 他にトランザクションコミット時のタイムスタンプを記録する

### タイムスタンプの理解

- タイムスタンプは実際に時刻そのものではなくシーケンス値を使用する
- 銀行を例にすると、
  - 振り込みが先か、引き落としが先かで随分と変わる（引き落としが先になると残高エラーの可能性とか）
    - この問題に対し、リアルでは受付番号を発行することで、同時に入ってきた客にも順番をつけることができる
- まさにこの発券機の役割をしているのがPDサーバが行なっているタイムスタンプ生成
  - TiDBではSELECTやINSERTなど処理の初めに必ずこのタイムスタンプを取得している
- タイムスタンプは、自国をベースに一意のシリアル値を組み合わせたもので、TSO(TimeStampOracle)と呼ばれる
  - TSOは、分散環境においては時刻の代わりを果たすもので、論理時間と呼ばれる
- トランザクション分離レベル
  - NewSQLの多くはSnapshot Isolationというもので、これはトランザクションの間は常に同じデータを読み込むことができる
    - 処理を開始した時点のスナップショットを見るような感じでデータを読み取る
    - DBはタイムスタンプより後に確定した変更は全て無視し、Undoログから復元して見せてくれる

## 5章 1つずつ壊してみよう

- ロードバランサー
  - 製品固有のロードバランサーとしてTiProxy
    - PDが収集しているクラスターの最新情報を取得し、タイムリーなリクエストのルーティングを可能にする
    - どのTiDBサーバーへの接続情報はPDからもらう
  - 自前でクラウドのロードバランサを立てる場合、pingやヘルスチェックエンドポイントで3回連続失敗で切り離すみたいなことをするので時間を要する
  - 以下の記事を見る限りだと、TiDB直接接続だとスケールインやメンテナンス時に接続断が発生するよう
    - [レバテックのTiDBにTiProxyを導入するまでにやったこと](https://zenn.dev/levtech/articles/enabling-tiproxy)
- TiDBサーバーを全て停止するとMySQLクライアントから接続できなくなる
  - 全て停止しなくても接続していたサーバーがスケールインされると接続が切れるので、アプリケーション側でも考慮が必要な点となる
- TiKVサーバー起動中の3台のうち1台(Leader)落としてみる
  - 3台のうち1台(Leader)を落としてもクエリ実行は成功する
  - TiKVサーバーは相互にハートビートを送っていて、リーダーの停止が確認できると、フォロワーノードのいずれかがリーダーに昇格して処理を続行する
    - 手動のクエリ実行だと↑を表現するのは中々難しいが、今回はTiDBからTiKVへのリクエストでリトライが行われたので、TiKVリーダーノードの停止は影響を受けない（リトライの分だけレイテンシは若干増加するが）
- さらにTiKVサーバーを残2台のうち1台を落としてみる
  - クエリは失敗するようになる
    - これはネットワーク分断障害と区別がつかないためこのような挙動となる
      - 自分が最新なのか、はたまたネットワーク分断で自分だけが切り離された古い情報なのかの判断がつかない
      - Raftアルゴリズムによる多数決の原理が関係している
- PDを停止してみる
  - PDもTiKVと同様に2台だと処理継続できるが、1台では不可
  - PDでもRaftアルゴリズムが使用されている

## 6章 NewSQLの仕組み・アーキテクチャを理解しよう

- コンピューティング層とストレージ層の分離

### コンピューティング層(TiDBノード)

- SQLパースと解析
  - クライアントから受け取ったSQL文を、字句解析(Lexer)と構文解析(Parser)を経て、AST(Abstract Syntax Tree)へと変換される
    - 字句解析(Lexer)
      - SELECTなどの最小単位のトークンに分割
    - 構文解析(Parser)
      - 切り分けたトークンをSQLの文法規約に照らし合わせ、親子関係を持つツリー構造(AST)に組み立てる
      - ここでSELECTの後にFROMがないみたいな構文エラーが発生する
- 最適化と実行計画の生成
  - ASTに対して、論理最適化・物理最適化が適用され、最適な実行計画を生成する
  - 論理最適化
    ```
    ここからがTiDBの腕の見せ所です。SQLの中身を「結果を変えずに、より効率的な手順」に書き換えます。

    - 述語プッシュダウン (Predicate Pushdown):
    JOIN した後に WHERE で絞り込むのではなく、JOIN する前に各テーブルで絞り込みを終わらせるよう順序を入れ替えます。転送データ量を最小化する、分散DBで最も重要な最適化です。

    - カラムプッシュダウン (Column Pruning):
    SELECT * と書いてあっても、実際に後の工程（JOINやORDER BY）で使わないカラムは、ストレージ（TiKV）から読み出さないように削ります。

    - サブクエリの相関解消 (Max/Min Eliminationなど):
    サブクエリを可能な限り JOIN に書き換えます。ループ（依存関係）をなくし、セットベースの処理に変えることで並列実行しやすくします。
    ```
  - 物理最適化
    ```
    論理的な手順が決まったら、次は**「どのインデックスを使い、どのアルゴリズムで結合するか」**を具体的に決めます。こちらは「コストベース最適化（CBO: Cost-Based Optimization）」と呼ばれます。

    TiDBは以下の要素を「コスト」として数値化し、合計が最小になるルートを探します。

    コストの構成要素:
    - CPU: データの比較やハッシュ計算の負荷。
    - Memory: ソートやハッシュテーブル作成に使うメモリ量。
    - Network: TiKVからTiDBへデータを送る通信負荷。
    - Disk I/O: ディスクから読み出すブロック数。

    具体的な選択肢の例：
    - Index Selection: WHERE a = 1 AND b = 10 のとき、インデックスAを使うか、Bを使うか、あるいはフルスキャン（TableScan）の方が速いかを、統計情報（データの分布）から予測される「読み取り行数」を元に判断します。
    - Join Algorithm Selection:
    - Hash Join: 片方のテーブルがメモリに収まるサイズなら、ハッシュテーブルを作って一気に結合します（大容量向け）。
    - Index LookUp Join: 外側のテーブルが小さく、内側のテーブルにインデックスがある場合に、1行ずつインデックスを引いて結合します（小〜中容量向け）。
    ```
  - この作業により、実際のデータアクセスや結合処理が効率化され、クエリの実行時間を短縮できる
- ステートレス
  - SQL変換のみでデータを保持したりしない、意図的にステートレスな設計となっている
  - なのでクライアントからの大量接続に対して容易にスケールアウトが可能

### ストレージ層(TiKVノード)

- データの永続化
  - ストレージエンジンはRocksDBを利用
  - Write-Ahead Log(WAL)やMemTable,SSTファイルといった仕組みでデータの永続性を保証する
    - RocksDBは、LSM-Tree
    - WAL: ディスク上のログファイルにこれから書くデータを追記(AppendOnly)
      - MemTableが揮発したとしてもリカバリ可能
    - MemTable: WALに書くと同時に、メモリ上のデータ構造であるMemTableにデータを展開する
      - SkipListという構造で、データが常にソートされた状態
    - SST(Sorted String Table)ファイル: MemTableがいっぱいになると一気にディスクに書き出す
      - 増えてくると裏側でコンパクション処理が走り、古いデータや削除フラグが立ったデータを掃除していく
    - →これらLSM-Tree（Log-Structured Merge-Tree）というアーキテクチャである
