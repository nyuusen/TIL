# コンテナセキュリティ コンテナ化されたアプリケーションを保護する要素技術

### コンテナへのシークレットの受け渡し

- コンテナイメージに埋めるのはビルド時にシークレットをハードコードする必要があるので良くない
  - シークレットの更新にもイメージ再ビルドが必要になってしまう
- 環境変数での受け渡しも良くない
  - クラッシュ時のデバックログに含まれたり、docker image inspectコマンドで確認可能
- マウントしたボリュームのファイルで受け渡すのが良い
  - tmpfsという仮想ファイルシステムに置かれる
    - これはメモリ上なのでコンテナ停止すれば消えるので安全
  - docker composeでは、tmpfsフィールドを使用する
    ```
    services:
      web:
        image: nginx:alpine
        tmpfs:
          - /app/cache:size=128M,mode=1777
    ```
- ネットワークインターフェイス越しの取得も安全
- 1つのホスト上のすべてのコンテナが同じカーネルを共有しているというのがセキュリティ上の意味がある
  - すべてのアプリケーションがすべてのシステムコールを必要とするわけではないので最小権限
    - カーネルのネットワークパケットフィルタリング技術であるBPF（Berkeley Packet Filter）を応用したseccomp-bpfがあり、これはプロセスが実行可能なシステムコールをホワイトリスト・ブラックリストで制限する機能
    - プロセスがシステムコールを発行するたびに、カーネル空間で高速にBPFフィルタの検証が行われる
    - Dockerやk8sではデフォルトでは40個以上危険なシステムコールをブロックしているが、さらにアプリケーションごとに最適化したカスタムプロファイルを適用することができる
  - ファイルパーミッション
    - Owner/Group/Otherのrwxビットだけではない
  - setuidとsetgid
