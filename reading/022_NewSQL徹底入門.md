## 第1章 なぜいまNewSQLが注目されているのか

### NewSQLの欠点

- レスポンスタイムの悪化
  - NewSQLに限らず分散システム全般が抱える課題だが、更新や参照の処理が走るたびにネットワークを介してノードの通信が発生する＝オーバーヘッドの発生
  - ただこの差はミリ秒単位のオーダーで収まるので、多くの採用実績も既にあるので、本当にレイテンシーを求めるシステムには不向きというくらいの話
- 若干割高
  - 同じリソース量であれば、RDBの方に軍配が上がることも
  - RDBではダメで、本当にNewSQLを導入すべきか？はよく検討が必要
- ノード数が多いので、障害遅延発生時の調査が難しい
  - 関連するコンポーネントが多いので切り分けが難しい
  - 近年はどの製品もマネージドサービス化が進んでおり、運用をベンダー側にオフロードできる
- 機能の制限と互換性の問題
  - ストアドやトリガー、特定のトランザクション分離レベルなど一部の機能が未対応で、DBMSの完全互換ではない
- 法規制とコンプラ対応
  - 地理分散の機能を持つ製品があり、その機能を利用するにはデータの分散配置や国際的なデータ移転に関する規制への対応を求められる
- 分析系の用途には向かない
  - OLTP向けに作れており、短い期間の多数のトランザクションを大量に処理することにフォーカスしている
  - 分析系のSQLをサポートしている製品も多いが、SnowflakeやBigQueryといった分析特化型のDBMSよりは性能は劣る

### 代表的なNewSQL製品

#### Spanner

- 分散トランザクションをサポートするために、原子時計やGPSといったハードウェアレベルでの最適化が行われている
- 垂直統合の特性上、GoogleCloudでしか動かない（他の製品はソフトウェアとして作られているので）
- マルチリージョン採用時のSLAが99.999%
- SQLは一部PostgreSQLとの互換性あり

#### TiDB

- 当初は中国で開発された後、米国に拠点を移動
- MySQLプロトコル互換（完全互換ではない）
- Apache License 2.0のオープンソース
- HTAP
- AWS, Azure, Google CloudでフルマネージドのDBaaSを提供

#### YugabyteDB

- 元Facebook社のエンジニアたちによって開発された
- Apache License 2.0のオープンソース
- PostgreSQL互換のSQL API(YSQL)とCassandra互換のNoSQL API(YCQL)をサポート

#### CockroachDB

- 元Google社員により開発されたSpannerに触発された製品
- マルチリージョン採用時にSLAが99.999%
- Cockroachはゴキブリの意味で、どんな障害にも耐えて生き残る堅牢性を象徴
- 標準的なSQLをサポートしており、既存のSQLベースのアプリケーションやツールとの互換性が高い

#### Aurora DSQL

- re:Invent 2024で発表された
- クエリ処理層・コンピュート層・ストレージ層がそれぞれ独立してスケーリングする
- シングルリージョンで99.99%, マルチリージョンで99.999%
- サーバーレス設計であり、パッチ適用・アップグレード・メンテナンスダウンタイムが不要
- 大きな特徴として、楽観的同時実行制御（OCC）を採用

### データベースの分類

- 単一インスタンス
  - 構成がシンプル
  - 更新/参照が同一マシンでキャッシュの問題もないし、システムクロックも単一
  - スケールアウトは不可なので、スケーラビリティに限界あり
  - HA構成をとるなどして障害発生時は数分後に復旧するみたいなイメージ（要件的に許容されないといけない）
- レプリケーション
  - 複数台稼働させて、データを複製する
  - プライマリとレプリカは固有のデータを保持する（クエリエンジンとストレージエンジンは分かれていない）
  - 読み取り処理はスケールアウトできるが、書き込み処理はスケールアウトしない
  - 同期レプリケーション（確実に同期されるがレイテンシが悪化）か非同期レプリケーション（レイテンシは良いがRPOは数秒になる）か
  - スケーラビリティと可用性の観点で完全な解決策ではないが、多くのRDBに組み込まれて設定も簡単で、現実的な選択肢になる
- 双方向レプリケーションによるマルチプライマリ
  - 複数台で全てプライマリなので書き込み性能向上が期待できるが限定的
  - 同じデータを同時更新した時に衝突が容易に発生する
- マネージドシャーディング
  - 書き込み・読み取りを担うコーディネータの下にDBがいくつか配置される構成
  - コーディネータの負荷が高くなりがち
  - データが分散配置されるので、範囲検索した時など、複数のDBノードから値を取ったりするなどのネットワークオーバーヘッドや計算コストがかかる
  - 複数シャードに跨った2PCが発生するので書き込み性能が劣化する
- ログ適用可能なストレージを用いたDB
  - クエリエンジンとストレージエンジンを分離
  - AuroraやGoogleのAlloy DB
  - 従来のRDBの書き込みは、メモリ更新→ログ書き込み→実際のデータの書き込みというWAL
  - ログ適用可能なストレージを用いたDBの書き込みは、
    - クエリエンジン：メモリ更新→ログをストレージに送信
    - ストレージエンジン：サーバーからログを受信したら可用性の確保に必要な複製が完了したら応答を返す→非同期でログを適用してディスク更新
  - メリット
    - クエリエンジンにほぼIOが発生せず、CPUを使う計算処理に特化できる＆逆にストレージエンジン側は高性能なストレージを配備できるなど、それぞれの役割に応じた最適な増強が可能
      - この点から、Auroraは従来のRDBから3倍程度の性能upしているらしい
  - デメリット
    - 書き込み性能がスケールしない
    - 各クラウドのログ適用可能なストレージには1クラスタあたり128TBという容量制限がある
- 分散ストレージ＋SQL
  - 読み取りも書き込みもスケールでき、可用性に優れている
  - サーバー間の通信が増えるため、レイテンシよりもスループットを重視したユースケース向け

## 第4章 NewSQLの要素技術

### ストレージエンジン

- 通常のRDBで採用されるB-treeでは、特に読み取り時に安定した性能を発揮するが、書き込み時にはランダムなディスクアクセスが増加し非効率となるケースがある
  - データの挿入や更新が発生するとページ分割が発生し、新しく作られたページはディスク上の別の領域に書き込まれる
  - 論理的には隣であっても、物理的には遠く離れたディスク領域に保存される
- そのため、書き込みヘビーなワークロードを扱うNewSQLでは、LSM-Treeというデータ構造を扱う

#### LSM-Tree

- Log-Structured Merge Tree
- 高い書き込み性能を実現、読み取り性能はB-treeに劣るケースがある
- コンポーネント構成は、MemTables, SSTable, マージ＆コンパクションプロセス
- MemTable
  - まず新たな書き込みが挿入される
  - 一定のサイズに到達するとSSTable(SortedStringTable)の形でディスクにフラッシュされる
- SSTable
  - ディスクに配置されるソート済みのデータファイル
  - 追記型（ファイルを更新しない）
  - シーケンシャルアクセスでディスクに書き込まれるため、ランダムアクセスのオーバーヘッドを回避
  - フラッシュの繰り返しでSSTableのファイル数が単調に増加していくことを避けるため、マージ＆コンパクションプロセスがバックグラウンドで実行される
- マージ＆コンパクション
  - 複数のSSTableファイルをマージし、1つの大きなSStableファイルを作成
    - この過程で保持する必要のないデータを削除および圧縮処理を行う
- B-treeとの比較
  - MemTableへの書き込みによる低いレイテンシ
  - 追記型のSSTableファイルのフラッシュによる高スループットな書き込みを実現
  - B-treeは固定長のブロック単位でデータを管理しているので、1つのレコードを更新する場合でもブロック全体をディスクに上書きする必要がある
    - この上書きのために、事前に対象のブロックを読み込んでおく必要があり、実際に更新したいデータ量に比べて多くのIOが発生する
    - LSM-treeは追記型のデータ構造なので、上記を回避している
      - マージ＆コンパクションプロセスにより、定期的に多くのIOが発生するが、このタイミングは調整可能
- RocksDB
  - LSM-treeの実装
  - これがNewSQLで使われる理由は、LSM-treeの追記型により、複数のバージョンを保持することができ、MVCCをストレージエンジンレイヤからサポートできる

### シャーディング

- データを複数の物理的なノードに分散させる
  - 単一のマシン内で複数のテーブルに分けるパーティショニングとは別
- シャードは、製品ごとに呼称が異なる
  - TiDBはリージョン
- シャーディングはキーの値に応じて、レコードを納めるシャードを決定する
  - この際のアルゴリズムに「レンジシャーディング」と「ハッシュシャーディング」の2種類がある
- レンジシャーディング
  - 範囲検索に強い
  - 一方で書き込み時にホットスポットが発生したり、読み取り時にも特定ノードに負荷がかかりやすい
  - SpannerやTiDBで採用
- ハッシュシャーディング
  - レンジシャーディングの逆のメリデメがある
- 自動シャーディング
  - NewSQLは負荷に応じてスケーリングするケースが多いため、手でシャーディングするのは現実的ではない
  - テーブルのサイズの伸縮に合わせて、自動的にシャードを分割したり結合したりする
- インターリーブ
  - 頻繁にJOINされるテーブルを同じSplitに配置するSpannerの機能

### Raft

- 基本機能はリーダーの選出とログ複製

### トランザクション分離レベル

- READ COMMITTEDとREPEATABLE READ
  - READ COMMITTEDで発生する問題
    - Non-repeatable Read: 同じトランザクション内で2回同じデータを読んだら、1回目と2回目で値が変わってしまう。
  - REPEATABLE READで発生する問題
    - ファントムリード: 範囲検索（例：10歳〜20歳を数える）をしたとき、さっきまでいなかった新しいデータが割り込んで「数」が増えてしまう
    - ファントム＝幻 幻のように1行現れるみたいな
  - Non-repeatable ReadとPhantom Readの違いは、既存行に影響があるか/新たに行が増えたりするか
- New SQLはスナップショットアイソレーション
  - なぜ高レベルの分離レベルなのか？
    - NewSQLはネットワークを介して複数のノードが存在するためノードごとに真実が異なる可能性がある(従来のRDBと異なる)よってカオスになりやすい
    - この分散環境の地獄を高い整合性とスピードで解決するのがRaftアルゴリズムであり、MVCCを実現

### 同時実行制御

トランザクション分離レベルは一貫性のある、ある状態(ゴール)、仕様を指していて、そのゴールの実装の手段の1つとなる同時実行時の動きを制御する(アルゴリズム)のが同時実行制御(楽観的(MVCC)/悲観的(ロック))\
楽観的だとロックをしないので高速、悲観的だとロックするので遅くなるけど一貫性が保たれる

- 悲観的 ロックして更新
  - 一貫性向上
  - 競合によるリトライなし
  - ロックするためパフォーマンス低下
    - デッドロックが発生することも
  - SELECT FOR UPDATE
- 楽観的 ロックせずに更新
  - パフォーマンス向上
  - 競合したらリトライが必要
    - 更新するタイミングで、元のデータが変わっているかを確認し、変わっていたらトランザクションは失敗となる
  - 書き込みが激しく衝突すると、リトライが多発して逆に効率が悪くなる
- RedisやDynamoDBなどのNoSQL製品は従来楽観的同時制御が一般的なアプローチ
  - これらのDBは分散処理を前提としており、可用性やスループットを重視する設計のため
  - ユースケースとしても、ユーザーセッション情報やキャッシュやログの保持といった競合の発生しにくい用途
- しかしNewSQLでは従来のRDBが担ってきた役割を担うので、悲観的同時制御もサポートする製品が増えている
  - Spannerは悲観的同時実行制御のみをサポートし、DSQLは楽観的同時実行制御のみをサポートしている
    - DSQLが楽観的同時実行制御のみをサポートする理由は、分散システムのユースケースのおいては更新の競合は稀にしか発生しないという前提に立っているから
  - TiDBはどちらもサポートしているが、現在は悲観的トランザクションモードがデフォルトになっている
    - 楽観的同時実行制御は、2PCを使用して行う
    - [TiDB Pessimistic Transaction Mode | TiDB Docs](https://docs.pingcap.com/ja/tidb/stable/pessimistic-transaction/)
  - この動きを見るとわかるのが、高スループットのWebサービスから整合性を重視する銀行の勘定系までカバーするユースケースが広がっている

#### 補足:MySQLはどうなっているか？

- 分離レベルのデフォルトはREPEATABLE READ
  - ファントムリードまで防ぐ
- 同時実行制御の仕組み
  - MVCC（楽観的アプローチ）
    - 通常のロックなしSELECT文で使用される
    - データを上書きするのではなく、変更前をUndoログに保存しておく
  - ネクストキーロック（悲観的アプローチ）
    - INSERT,UPDATE,DELETEやSELECT FOR UPDATEなど明示的にロックが必要な場合に使用される
    - 対象の行だけでなく、その隙間の行に新しい行が入るのも防ぐ

### 分散トランザクション

- 一般に分散システムのおいて、ノード間の時刻というのは微妙にずれている
  - すなわち、すべてのノードが一致する絶対時間というものが存在しない
  - この時刻のずれををクロックスキュー(Clock Skew)という
- つまりTxAとTxBの順序性の問題が生じることになる
- Spanner
  - 原子時計とGPSというハードウェアを使っている
  - 最小時間・最大時間をとって、最大時間まで確実に経過することを待ってからコミットする動きになる
    - この両者の時間幅を限りなく少なくするために、上記のハードウェアを使っている
- 後発のTiDBやCockroachDBなど
  - 特殊なハードウェアを使わずに、一般的なNTPプロトコルを使用している
- タイムスタンプベースの同時実行制御
  - タイムスタンプオラクル
    - タイムスタンプを中央集権型機構で管理
    - タイムスタンプを発行するのはリーダーだけ（PDに問い合わせる）
    - TSOは、物理時間と論理時間からなる64ビット数値

## 第6章 NewSQLの標準SQLへの対応状況

### 主キー

- NewSQLでは、連番のように連続的に増加する値を主キーで使ってしまうと、特定のシャードに書き込みが集中するホットスポットが発生しやすい
- 主キー候補
  - UUID v4
    - 各社UUIDを生成する関数をサポートしている
    - v7を使いたい場合は基本的にはアプリケーション側で発行する
    - UUIDv4の欠点
      - 16バイトなのでサイズが大きい。インデックスのサイズも大きくなるので結合や検索時の性能劣化の懸念
        - ページ分割
      - 可読性が悪く目視でのデバッグが困難。URLなどでの視認性も悪い。
      - 順序性を持たないのでランダムに配置されキャッシュヒット率が悪い
  - ハッシュ
    - UUID以外でシャーディングキーをランダム性を担保するための手段として、シャーディングキーにハッシュ化した値を使う方法もある
    - 以下のケースで有効
      - 挿入・更新が多く高スループットが求められる
      - ホットスポットを回避したい
      - レンジスキャンよりも主キーによる一意検索が基本のワークロード
    - 欠点
      - 実装依存なので移植性が下がる
      - レンジスキャンが非効率

## 第7章 NewSQLのユースケース

- スケーラブルRDB
  - Write-heavyのシステムに最もよくフィットする
    - 具体的にはECの注文処理や金融の決済、IoT（IoTはあまり事例がないらしい）
- レバテックの例
  - 各マイクロサービスで持つデータストアの管理工数増が課題→TiDB導入
    - 複数のサービスのデータストアをまとめるとノイジーネイバー問題が発生する
      - それに対し、TiDBにはリソース制御機能（クエリ/ユーザーやユーザーグループ単位）でリソース使用量（CPU/メモリ/ストレージ帯域を制御）がある
    - サービスを跨るトランザクションをアプリケーション側で実装する必要がなくなった
      - Sagaパターン: イベントを数珠繋ぎ的に連鎖させていく。失敗時は補償トランザクションを発行。
