# シャーディングとパーティショニング

## シャーディング

- データを物理的に異なるDBサーバーに分散させる
- 物理的にDBが分かれるのでアプリ層やプロキシ層で「ユーザーAのデータはDBサーバー1」といった具合にルーティングする必要がある
  - アプリ層でやるならIDの値を2で割った余りで振り分けるとか
  - プロキシ層でやるならVitessやProxySQLを使うなど
  - SpannerやTiDBなどのNewSQL系は同等の機能をそもそも内蔵していたりとか
- 負荷分散とスケーラビリティで効果がある

## パーティショニング

- 1つの大きなテーブルを内部的にいくつかの小さな部屋に分ける
- アプリからは1つのテーブルに見える
  - DBエンジン側で適切や部屋に探しに行ってくれる
- 運用や検索の効率化
  - logテーブルを月毎にパーティション分けする
- CREATE TABLE文で、PARTITION BYで指定する

### 物理的な構造

- 通常MySQLでusersテーブルを作成するとusers.ibdファイルが1つ作られるが、パーティション化するとusers#P#p2025.ibdのようなファイルに分割される
- MySQLサーバはクエリを受け付けると、SQLレイヤでどのibdファイルを開くべきかを判断し、必要なファイルだけをスキャンする

## 補足

- TiDBのリージョンは、概念的にはシャーディングに近い（異なるTiKVに格納されるので）
